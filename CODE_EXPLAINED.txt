KEY PIECE 1: THE RETRIEVER
================================================================================

File : run_baselines.py  (also used in run_phi3_quick.py)
What : Takes a medical question, finds the 3 most relevant contexts from the
       500-document PubMedQA corpus using semantic similarity.

WHY THIS MATTERS
----------------
Without a retriever, the language model has to answer purely from memory —
which is a hallucination risk. By retrieving relevant passages first, we give
the model real facts to reference. This is the "R" in RAG.

THE CODE
--------

    from sentence_transformers import SentenceTransformer
    from sklearn.metrics.pairwise import cosine_similarity
    import numpy as np

    # Load the embedding model once at startup
    retriever_model = SentenceTransformer('all-MiniLM-L6-v2')

    # Embed ALL 500 contexts upfront (384-dimensional vectors each)
    contexts = df['context'].fillna("").tolist()
    context_embeddings = retriever_model.encode(contexts, show_progress_bar=False)

    def retrieve_contexts(question, k=3):
        """
        Retrieve top-k contexts for a question using cosine similarity.
        Returns: (contexts_str, top1_similarity, mean_similarity)
        """
        question_emb = retriever_model.encode([question])

        # Cosine similarity between the question and every context
        similarities = cosine_similarity(question_emb, context_embeddings)[0]

        # Sort by similarity score, highest first, take top k
        top_k_indices = np.argsort(similarities)[::-1][:k]
        top_k_sims    = similarities[top_k_indices]

        retrieved_context_texts = [contexts[idx] for idx in top_k_indices]
        concatenated = " | ".join(retrieved_context_texts)

        top1_sim = float(top_k_sims[0])
        mean_sim = float(np.mean(top_k_sims))

        return concatenated, top1_sim, mean_sim

WHY COSINE SIMILARITY (NOT FAISS)
----------------------------------
The original version used FAISS (Facebook AI Similarity Search). FAISS
threw this error on the project environment:

    ModuleNotFoundError: numpy._core

FAISS depended on an internal NumPy API (_core) that was removed in NumPy 2.x.
The fix was to replace FAISS with sklearn's cosine_similarity — which does the
same thing (dot product of normalised vectors) without the version conflict.
At 500 documents the speed difference is negligible; FAISS is optimised for
millions of vectors.

The similarity score (0.0–1.0) is also stored in each result row so retrieval
quality can be analysed separately from generation quality.

PHI-3 VERSION — CONTEXT TRUNCATION
-------------------------------------
In run_phi3_quick.py, each retrieved passage is capped at 150 characters:

    def retrieve(q):
        scores = cosine_similarity(enc.encode([q]), embs)[0]
        top3   = np.argsort(scores)[::-1][:3]
        # keep only first 150 chars per passage to avoid very long prompts
        return " | ".join(ctxs[i][:150] for i in top3)

This was essential for CPU inference — full PubMed abstracts (~1500 tokens
each) caused Phi-3 Mini to time out on the home machine. Truncating to 150
chars per passage reduced prompt length from ~4500 tokens to ~450 tokens,
making RAG completions tractable on CPU.

KEY PIECE 2: GENERATION FUNCTIONS (LOCAL MODELS)
================================================================================

File : run_baselines.py
What : Takes a question (and optionally a retrieved context), builds a prompt,
       and runs it through a local Hugging Face model to get a yes/no/maybe
       answer.

THE CODE (FLAN-T5 — same pattern used for BioBART and DistilBART)
------------------------------------------------------------------

    def generate_flan_t5(question, context=None, use_context=True):
        """
        Generate answer using Flan-T5.
        Returns: (answer_text, token_length)
        """
        if use_context and context:
            prompt = f"""Based on the following medical context, answer the \
question with yes, no, or maybe, followed by brief reasoning.

Context: {context}

Question: {question}

Answer:"""
        else:
            prompt = f"""Answer the following medical question with yes, no, \
or maybe, followed by brief reasoning.

Question: {question}

Answer:"""

        inputs = flan_tokenizer(
            prompt,
            return_tensors="pt",
            max_length=512,
            truncation=True
        ).to(DEVICE)

        with torch.no_grad():
            outputs = flan_model.generate(
                **inputs,
                max_length=150,
                min_length=5,
                num_beams=4,        # beam search for better output quality
                early_stopping=True,
                temperature=0.7,
                do_sample=False     # deterministic — same input = same output
            )

        answer_text  = flan_tokenizer.decode(outputs[0], skip_special_tokens=True)
        token_length = len(outputs[0])
        return answer_text, token_length

KEY DESIGN DECISIONS
---------------------
- Same function signature for all three local models (FLAN-T5, BioBART,
  DistilBART). The experiment loop calls them identically, which makes adding
  or removing models simple.
- use_context=True/False controls whether the RAG prompt or the No-RAG prompt
  is built. The context itself is passed in but only inserted when use_context
  is True.
- do_sample=False makes generation deterministic (reproducible results across
  runs with the same inputs).
- max_length=512 on the input side prevents tokeniser errors on very long
  PubMed abstracts.
- token_length stored per row allows analysis of whether longer answers
  correlate with any accuracy pattern.

WHICH MODELS RAN
-----------------
Model                         | Load method           | Notes
------------------------------|------------------------|----------------------------
google/flan-t5-base           | AutoModelForSeq2SeqLM | Always loaded, 250M params
GanjinZero/biobart-base       | AutoModelForSeq2SeqLM | PubMed pre-trained BART
sshleifer/distilbart-cnn-12-6 | AutoModelForSeq2SeqLM | Distilled BART, CNN-trained
google/t5-v1_1-base           | — FAILED —            | Tokeniser error, dropped

All three are loaded with a try/except block; if any model fails to download
or initialise, a flag (e.g. biobart_available = False) is set and that model's
conditions are skipped cleanly.

KEY PIECE 3: GPT-4 INTEGRATION WITH CONDITIONAL EXECUTION
================================================================================

File : run_baselines.py, run_gpt_only.py
What : Calls the OpenAI API to generate answers, but ONLY if an API key is
       present. If no key, the GPT conditions are skipped entirely — no fake
       data, no crash.

THE CODE
--------

    # Read API key from environment variable — NEVER hardcoded
    gpt_api_key = os.getenv("OPENAI_API_KEY")

    def generate_gpt(question, context=None, use_context=True,
                     debug=False, sample_id=None):
        """
        Generate answer using GPT-4 API.
        Returns: (answer_text, token_length)
        """
        if use_context and context:
            user_msg = f"""Based on the following medical context, answer the \
question with yes, no, or maybe, followed by brief reasoning.

Context: {context}

Question: {question}

Answer:"""
        else:
            user_msg = f"""Answer the following medical question with yes, no, \
or maybe, followed by brief reasoning.

Question: {question}

Answer:"""

        try:
            from openai import OpenAI          # v1 SDK (openai >= 1.0.0)
            client  = OpenAI(api_key=gpt_api_key)
            response = client.chat.completions.create(
                model      = "gpt-4",
                messages   = [{"role": "user", "content": user_msg}],
                max_tokens = 150,
                temperature= 0.7
            )
            answer_text  = response.choices[0].message.content
            token_length = len(answer_text.split())
            return answer_text, token_length

        except Exception as e:
            error_msg = str(e)[:50]
            return f"[GPT Error: {error_msg}]", 5    # safe fallback

    # In the experiment setup — GPT conditions added only if key exists:
    if gpt_api_key:
        conditions.extend([("gpt", "no_rag"), ("gpt", "rag")])
    else:
        print("⚠ Skipping GPT: missing OPENAI_API_KEY environment variable")
        print("  Set key with: $env:OPENAI_API_KEY = 'your-key-here'")

PROBLEMS ENCOUNTERED WITH GPT
-------------------------------
1. openai not installed in the virtual environment
   Error: [GPT Error: No module named 'openai']
   Fix  : pip install openai
          The package was listed in requirements but never installed into .venv.

2. Deprecated v0 API syntax
   Old code (broken in openai >= 1.0.0):
       openai.ChatCompletion.create(...)
   New code:
       client = OpenAI(api_key=...)
       client.chat.completions.create(...)
   The v1 OpenAI SDK introduced a breaking API change; all call sites had
   to be updated.

3. HTTP 429 — Quota exhausted
   Despite valid authentication, every API call returned:
       Error code: 429 — You exceeded your current quota
   This happened with two separate API keys (both had zero billing credit).
   The try/except block caught all 429s gracefully; every row was still
   written to CSV (with "[GPT Error: ...]" in the answer column).
   The quota-limited run in results_gpt/ (16% accuracy) represents 100%
   API failures, not real model performance.

4. API key security incident
   An API key was accidentally pasted into plain text. It was immediately
   revoked at platform.openai.com/account/api-keys.
   Keys are now stored only as environment variables:
       $env:OPENAI_API_KEY = "sk-proj-..."
   Never hardcoded, never pasted in plain text anywhere.

KEY PIECE 4: LABEL EXTRACTION (YES / NO / MAYBE PARSING)
================================================================================

File : run_baselines.py, run_phi3_quick.py
What : Parses the raw generated text (which might be a full sentence or
       paragraph) and extracts one of three class labels: yes, no, maybe.

THE CODE (from run_baselines.py)
---------------------------------

    def extract_label(answer_text):
        """
        Extract yes/no/maybe from generated answer text.
        Simple heuristic: look for first occurrence of yes/no/maybe.
        """
        text_lower = answer_text.lower()

        if 'yes' in text_lower:
            return 'yes'
        elif 'no' in text_lower:
            return 'no'
        elif 'maybe' in text_lower or 'perhaps' in text_lower \
             or 'uncertain' in text_lower:
            return 'maybe'
        else:
            # If none found, check the very first word
            first_word = text_lower.split()[0] if text_lower.split() else ""
            if first_word in ['yes', 'no', 'maybe']:
                return first_word
            else:
                return 'maybe'   # conservative default

THE SIMPLIFIED VERSION (run_phi3_quick.py)
-------------------------------------------

    def label(text):
        t = text.lower()
        if "yes"   in t: return "yes"
        if "no"    in t: return "no"
        if "maybe" in t or "perhaps" in t: return "maybe"
        return "maybe"

WHY A HEURISTIC INSTEAD OF SOMETHING FANCIER
----------------------------------------------
The models were not fine-tuned on PubMedQA's exact label schema. They
generate natural language answers like:

    "Yes, the study supports this conclusion because..."
    "No, there is no evidence that..."
    "It is uncertain whether..."

A keyword scan is simple, fast, and surprisingly effective for this use case.
The default of 'maybe' is conservative — when the model's answer is genuinely
ambiguous, defaulting to 'maybe' is less likely to introduce false positives
for either 'yes' or 'no'.

KNOWN LIMITATION
-----------------
The word "no" appears in many English words (notable, knowledge, cannot, etc.).
This means some answers are mis-classified as 'no' when they should be 'maybe'.
A more robust approach would check for the label as a standalone word using
regex (e.g. r'\bno\b'), but the current version was kept simple for the
baseline and the effect is consistent across all model conditions (so
comparisons between conditions are still fair).

WHY THIS STRUCTURE
-------------------
Having a single flat loop over (model, setup) pairs means:
- Adding a new model requires only: adding a condition tuple + a generate_X()
  function. Nothing else changes.
- Every result goes into the same DataFrame, making cross-model comparisons
  trivial (just filter by the 'model' column).
- The run_id ties all rows from one execution together, so if I run the script
  multiple times the results can be distinguished.

THE DYNAMIC MODEL LIST FIX
----------------------------
After BioBART replaced T5-v1_1, the metrics section still had:

    if t5_v1_1_available:       # <-- NameError: this variable no longer existed
        model_list.append(...)

This crashed AFTER all 800 experiment rows had generated, just before writing
the metrics CSV. All generation work was lost.

Fix — build the model list from the data instead of from flags:

    # Before (brittle — breaks when models change):
    model_list = ['flan_t5', 'biobart']
    if gpt_api_key: model_list.append('gpt')

    # After (robust — works for any models that actually ran):
    model_list = results_df['model'].unique().tolist()

KEY PIECE 6: METRICS COMPUTATION & OUTPUT
================================================================================

File : run_baselines.py (Steps 7–11)
What : Groups results by (model, retrieval_setup), computes accuracy / macro F1
       / precision / recall for each group, saves four output CSV files.

THE CODE
--------

    from sklearn.metrics import (
        accuracy_score, f1_score,
        precision_score, recall_score, confusion_matrix
    )

    metrics_list = []

    # Dynamic model list — works regardless of which models ran
    model_list = results_df['model'].unique().tolist()

    for model in model_list:
        for setup in ['no_rag', 'rag']:

            subset = results_df[
                (results_df['model'] == model) &
                (results_df['retrieval_setup'] == setup)
            ]
            if len(subset) == 0:
                continue

            y_true = subset['true_label'].values
            y_pred = subset['prediction_label'].values

            accuracy  = accuracy_score(y_true, y_pred)
            macro_f1  = f1_score(y_true, y_pred, average='macro', zero_division=0)
            macro_prec= precision_score(y_true, y_pred, average='macro', zero_division=0)
            macro_rec = recall_score(y_true, y_pred, average='macro', zero_division=0)

            metrics_list.append({
                'model':            model,
                'retrieval_setup':  setup,
                'num_samples':      len(subset),
                'accuracy':         round(accuracy,   4),
                'macro_f1':         round(macro_f1,   4),
                'precision_macro':  round(macro_prec, 4),
                'recall_macro':     round(macro_rec,  4),
            })

    # Write the summary CSV
    metrics_df = pd.DataFrame(metrics_list)
    metrics_df.to_csv(OUTPUT_DIR / "metrics_summary.csv", index=False)

    # Confusion matrices (one entry per true/predicted label pair per condition)
    confusion_rows = []
    for model in model_list:
        for setup in ['no_rag', 'rag']:
            subset = results_df[
                (results_df['model'] == model) &
                (results_df['retrieval_setup'] == setup)
            ]
            if len(subset) == 0:
                continue
            labels = sorted(set(subset['true_label']) | set(subset['prediction_label']))
            cm = confusion_matrix(subset['true_label'], subset['prediction_label'],
                                  labels=labels)
            for i, pred_lbl in enumerate(labels):
                for j, true_lbl in enumerate(labels):
                    confusion_rows.append({
                        'model': model, 'retrieval_setup': setup,
                        'true_label': true_lbl, 'predicted_label': pred_lbl,
                        'count': int(cm[j, i])
                    })

    pd.DataFrame(confusion_rows).to_csv(OUTPUT_DIR / "confusion_matrices.csv", index=False)

WHY MACRO-AVERAGE F1
---------------------
The PubMedQA label distribution is imbalanced (more 'yes' than 'no'/'maybe').
Macro-average F1 treats each class equally regardless of frequency — it does
not reward a model just for predicting the majority class every time.
This makes it the most informative single metric for comparing models on an
imbalanced 3-class problem.

FOUR OUTPUT FILES PER RUN
--------------------------
1. all_experiments.csv    — every single prediction (200–800 rows)
2. metrics_summary.csv    — one row per (model, condition)
3. confusion_matrices.csv — error breakdown by true vs predicted label
4. prediction_distributions.csv — % of yes/no/maybe per condition

These are all saved to a versioned output directory (results/, results_v2/,
results_v3/) so older runs are preserved for comparison.
